Pra fundação técnica. Motivo: custo de tokens. OpenClaw bebe muito em dev pesado. Cloud Code roda na VPS Hostinger via SSH — mais barato pra build.

Cloud Code SSH na VPS · executa PRD em 4 chunks: segurança · branding · escritório · deploy. ~1h30 de execução total.

Frase-mãe da A14: Mission Control não é dashboard que você decora. É workflow conversacional onde a Amora constrói com você — você manda áudio na academia, ela manda print, vocês iteram. 85% do trabalho em 1 prompt. O resto é refinamento.

Adaptações Managed

2 caminhos pra você fazer.

Aula original mostra workflow com Cloud Code + VPS — preferência de produção do Bruno (custo de tokens). Aluno Managed do curso V2 tem 2 caminhos válidos. Ambos funcionam. Bruno escolheu B por motivo de custo. Aluno PME que quer simplicidade fica em A.

Caminho Como Trade-off
A — Tudo via agente Managed (default V2) Pede pra agente executar PRD direto, dentro do workspace OpenClaw Mais lento em token · sem necessidade de VPS / Cloud Code / Tailscale
B — Aluno avançado com VPS (gancho R1-R4) Cloud Code SSH na VPS + executa PRD via Cloud Code Mais barato em tokens · setup inicial maior

Princípio: ambos os caminhos chegam no mesmo lugar. Se você é PME e quer agente pra você — caminho A. Se você tá disposto a montar VPS + Tailscale + Cloud Code pra economizar tokens em build pesado — caminho B (e os bônus técnicos R1-R4 cobrem).

Stack opcional

Métricas de redes — Apify ou alternativas.

Mission Control do Bruno usa Apify (US$ 30-40/mês) pra puxar métricas de LinkedIn, Instagram, YouTube, Twitter. Se você não quer pagar Apify desde o day 1, tem alternativas gratuitas:

✓ LinkedIn — API oficial limitada · scraping pelo browser via skill custom (gratuito mas frágil)

✓ Instagram — Instagram Graph API oficial (gratuita pra business · setup 30min)

✓ YouTube — YouTube Data API oficial (gratuita até 10k requests/dia · sobra pra PME)

✓ Twitter/X — API paga (US$ 100/mês básico) OU manual export semanal

✓ Apify — paga (US$ 30-40/mês) · pega tudo isso num dashboard único · vale quando você quer setup < 1h

Recomendação Bruno: começa com APIs gratuitas pra YouTube + Instagram. Se travar e quiser setup unificado, paga Apify (US$ 30-40/mês cobre tudo). Não precisa decidir agora.

Princípios A10 aplicados

Segurança em cada etapa.

A14 não ensina segurança nova — aplica os princípios da A10 num caso real. Cada etapa do build do Bruno tem proteção embutida:

1

Audit antes de clonar repo de terceiro

Etapa 3 — Amora roda audit no tenacitOS antes de qualquer commit. Procura prompt injection, malware, vulnerabilidades. Repo de terceiro entra com desconfiança, não com fé.

2

Scope mínimo por agente

Cada agente que toca no Mission Control tem escopo restrito. Cloud Code tem permissão de executar PRD na VPS, mas não tem credenciais de produção. Princípio do menor privilégio.

3

Row-level security (RLS) no Supabase

Dashboard multi-tenant exige RLS por default. Cada usuário só vê dados próprios. Esquecer RLS = vazamento horizontal de dados.

4

Defesa contra prompt injection no dashboard

Se o dashboard recebe input do usuário e passa pra agente, sanitização é obrigatória. User input ≠ instrução pro agente. Recap A10 — separação de canal humano e canal de instrução.

Prompts prontos pra colar

4 prompts do cheatsheet mission-control.

Resumo dos prompts canônicos do mission-control.md. Cola direto no agente principal — ele conduz a partir daí, replicando o workflow do Bruno.

Prompt #1 · pesquisar template cola no agente principal
Quero construir um Mission Control customizado pro meu negócio. Contexto: [DESCREVE TEU NEGÓCIO + 3-5 INFORMAÇÕES QUE QUER VER NUM DASHBOARD ÚNICO — ex: kanban de tasks, métricas de redes, leads novos, financeiro do mês]. Pesquisa pra mim 3-5 templates open-source no GitHub que servem de base (stacks: Next.js + Tailwind + shadcn/ui · React + Supabase). Critérios: 1. Manutenção ativa (commits últimos 6 meses) 2. Licença permissiva (MIT, Apache) 3. Estrutura modular (fácil adaptar pro meu caso) 4. Componentes que cobrem 70%+ do que preciso (não vou reinventar) Retorna: nome do template · URL · stack · % de aderência ao meu caso · princípio recomendação. Quero comparativo direto, não lista genérica.
💡 Bruno achou TenacityOS desse jeito (Carlos Azaustre, GitHub). Princípio: não reinventar a roda. Pesquisa primeiro, adapta depois.
Prompt #2 · audit de segurança no template cola no agente principal
Antes de clonar o repo [URL DO TEMPLATE] pro meu workspace, faz audit de segurança completo. Procura especificamente: 1. Prompt injection em strings de prompts (se o template tem chamada pra LLM) 2. Dependências com vulnerabilidades conhecidas (npm audit · CVEs) 3. Hardcoded secrets ou tokens (mesmo que comentados) 4. Endpoints sem auth (rotas públicas que não deveriam ser) 5. Acesso a localStorage/sessionStorage de domínios externos 6. Scripts inline ou eval() suspeito 7. Network calls pra domínios não-canônicos do projeto Relatório executivo: severidade (crítico/alto/médio/baixo) · onde tá · o que faz · ação recomendada (não-clonar / fix-antes-de-rodar / revisar / aceitar). Repo de terceiro entra com desconfiança, não com fé.
💡 Princípio A10 vivo. Bruno fez exatamente isso com TenacityOS. Audit antes de qualquer commit pode salvar você de prompt injection ou malware oculto.
Prompt #3 · montar PRD com superpowers cola no agente principal
Quero montar PRD completo pro meu Mission Control, baseado no template [URL] que já passou audit de segurança. Usa skill superpowers (brainstorming + writing-plans + executing-plans) pra fazer revisão profunda. Eu quero perguntas críticas, não checklist genérico. Frame: 1. Brainstorming: 5-7 perguntas que NÃO posso deixar de responder antes de executar (escopo · permissões · fork vs upstream · branding · comportamento dos personagens · hospedagem · custos) 2. Writing-plans: organiza em must-have vs nice-to-have. Corte agressivo no nice-to-have — só fica o que é bloqueante pro V1 3. Executing-plans: divide em chunks executáveis (4-6 chunks de 30-45min cada) pra eu rodar via Cloud Code OU OpenClaw Output: PRD em Markdown · pronto pra colar como contexto na execução · sem fluff, só o que vai virar código.
💡 Skill superpowers faz a diferença aqui. Sem ela, PRD vira lista de wishlist. Com ela, vira documento executável.
Prompt #4 · iterar via Telegram cola no agente principal
PRD executado · Mission Control rodando em [URL DEV LOCAL ou STAGING]. Agora quero iterar via áudio no Telegram, modo Bruno na academia. Como vai funcionar: 1. Eu mando áudio descrevendo o que quero ajustar (ex: "deixa o card de Kanban no topo · move métricas pra direita · cor do badge amarelo tá feio") 2. Tu transcreve mentalmente · interpreta · executa 3. Tu manda PRINT do resultado pra mim (screenshot do dashboard depois da mudança) 4. Se precisa de validação ANTES de executar (mudança grande), pergunta 5. Se é mudança visual menor (tipo reordenar, mudar cor), executa direto e me mostra o resultado Princípio: 85% do trabalho em 1 prompt. Resto é refinamento via áudio. Não pergunta a cada microajuste — só quando muda escopo materialmente.
💡 Esse é o coração da A14. Bruno fez exatamente isso na academia, recebendo prints da Amora e voltando com áudio. Workflow conversacional puro.
Proteção anti-overengineering

Quando NÃO construir Mission Control.

A14 mostra build complexo — não é pra todo mundo. Aluno PME dia-1 NÃO precisa de Mission Control. Sinais de que ainda não é a hora:

Sintoma Diagnóstico
Você ainda não usa o agente todo dia Construa relação primeiro · cockpit é refinamento
Você tem 1 agente só, sem multi-agente ativo Telegram já é cockpit suficiente · Mission Control vira gold-plating
Você não tem >5 fontes de dados pra unificar Spreadsheet ou Notion resolvem com 10% do trabalho
Você não vai mexer no dashboard toda semana Setup pesado, manutenção zero = código morto em 60 dias
Você só quer porque viu o do Bruno Estética > função. Volta pro básico

Princípio: Mission Control é refinamento avançado. Maioria dos PME no day 30-60 ainda tá calibrando agente principal — esse é o ROI real. Mission Control faz sentido quando você tem 1 OU mais agentes operando 24/7 com múltiplas fontes de dados que precisam de visão unificada. Se não é teu caso ainda, deixa pra depois sem culpa.

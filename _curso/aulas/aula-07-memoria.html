Aprofundamento do Bloco 1

Mecânica primeiro — como ele escreve.

Antes de teoria. Como agente OpenClaw escreve memória no day a day — e onde você participa.

Daily notes — fluxo automático

Toda atividade significativa do dia é capturada em memory/YYYY-MM-DD.md. Markdown puro, padrão OpenClaw. Solta na raiz de memory/ (recap A6, não em sessions/).

✓ Quando agente escreve: ao longo da conversa, no final do dia, ou quando você pede explicitamente

✓ O que captura: decisões, calls, pendências, contexto novo, resultados de tarefas executadas

✓ Formato: bullet points + timestamp · não enche linguiça · captura essência

✓ Quem lê depois: próprio agente no boot do dia seguinte · você quando consulta

Salvamento direcionado — quando você ajuda

Você pode pedir explicitamente que ele salve algo importante. E pode direcionar onde:

Prompt · salvamento direcionado cola no Telegram
salva isso importante: [DECISÃO / CONTEXTO] Salva tanto na daily note de hoje quanto no lugar estruturado que faz sentido (ex: pasta do projeto, context/decisoes/, ou context/people/{nome}.md se for sobre alguém específico). Me mostra onde você salvou.
💡 Use sempre que decidir algo que você vai querer consultar daqui a 3 meses. Frase-âncora: "se eu vou perguntar isso de novo, salva."

Memory Flush — feature nativa OpenClaw

Conceito técnico importante: antes da compactação automática disparar, agente faz um "silent memory flush" — tenta salvar nas pastas certas antes de comprimir. Documentado em docs.openclaw.ai/concepts/compaction.

Confiabilidade real (estimativa Bruno): ~70% das vezes funciona limpo. Compactação às vezes mata o salvamento — token budget acaba antes do flush completar. Por isso você ajuda salvando manual quando o assunto importa. Não confia 100% no flush automático.

Aprofundamento do Bloco 2

3 tipos de memória trabalhando juntos.

Agente OpenClaw tem 3 sistemas de memória que conversam entre si. Cada um cobre coisa diferente. Não substituem — complementam.

Tabela comparativa

Tipo O que cobre Quando ativa
1. Padrão (boot) MEMORY.md (long-term) + daily notes 48h + MAPAs Toda nova sessão · automático
2. Sessão Chat atual no Telegram/Gateway Enquanto você conversa · janela 200k tokens
3. Semântica Busca por significado no workspace inteiro Quando você pergunta · usa embeddings (API key A1)

1. Memória padrão · como ele acorda

Quando agente "acorda" numa nova sessão (você abre o Telegram, manda mensagem nova depois de horas/dias), ele automaticamente:

✓ Lê MEMORY.md — long-term self-knowledge (recap A6)

✓ Lê daily notes de hoje + ontem — 48h de contexto (padrão OpenClaw)

✓ Carrega MAPAs do workspace — sabe onde tá cada tipo de coisa (Princípio 12, recap A6)

✓ Resultado: ele acorda já sabendo de tudo que tá rolando — não começa do zero

2. Memória de sessão · chat atual

Janela do contexto da conversa em curso. Tem limite — 200k tokens com GPT-5.5 (~150 páginas de texto). Cobre Bloco 3 abaixo.

3. Memória semântica · busca por significado

O sistema mais poderoso. Cobre Bloco 4 abaixo. Já vem ativada quando você configurou OpenAI API key em A1 — não precisa skill nova nem setup adicional.

Por que precisava da API key na A1: embeddings (vetores numéricos que representam significado) rodam via OpenAI direto. OAuth da Fase B cobre raciocínio · API key da Fase A cobre Whisper + DALL-E + embeddings da memória semântica. Cada credencial tem papel específico.

Aprofundamento extra · contexto quente

memory/hot.md — o resumo do agora.

Os 3 tipos cobrem boot, sessão e busca histórica. Mas tem uma quarta peça que vive na raiz de memory/ e merece nome próprio: hot.md. Pensa nele como a tela inicial do agente — snapshot do que importa agora, atualizado o tempo todo, lido em toda sessão.

O que tem dentro

✓ Top prioridades da semana — 3-5 itens que precisam de atenção real

✓ Negociações ativas — quem tá pendurado, último toque, próximo passo

✓ Decisões recentes — o que foi fechado nos últimos 7-14 dias

✓ Métricas-chave — números do mês que importam (receita, alunos, eventos)

✓ Prazos críticos — eventos com data próxima que não podem passar

Quem mantém

hot.md é auto-gerenciado. O agente atualiza no fluxo normal: quando você decide algo importante, surgem prioridades novas, ou um prazo se aproxima — ele edita. Você empurra atualização explícita com a skill /salve (cola final de sessão · agente lê o que aconteceu e atualiza pendências, decisões, prazos, hot). Você não escreve hot.md à mão.

Função 1 · boot context

Toda nova sessão, agente lê hot.md junto com MEMORY.md e dailies. Diferença: MEMORY.md é durável (auto-conhecimento que muda devagar), dailies são cronologia (o que aconteceu), hot.md é estado atual (o que importa agora). Ele entra no flow já sabendo o que tá quente — não precisa perguntar "no que a gente tava?".

Função 2 · combustível do heartbeat

Aqui é onde hot.md conecta com A9. Heartbeat (mecanismo de proatividade · poll periódico) consulta hot.md a cada batida pra decidir o que vigiar. Sem hot.md, heartbeat fica perdido — checa coisa genérica que não importa pra você nessa semana.

Exemplo concreto: você fechou contrato com a Furia, prazo do primeiro entregável é 22/04. Skill /salve roda · agente atualiza hot.md com "prazo crítico Furia 22/04". A cada heartbeat (a cada 30min), agente lê hot.md · vê o prazo · checa se passou ou tá próximo · te avisa só se valer interromper. Sem hot.md, heartbeat não saberia que esse prazo existia.

Distinção importante (recap A3): HEARTBEAT.md (raiz do workspace) é a configuração do mecanismo — poll period, checks rotacionáveis, quiet hours. memory/hot.md é o contexto que esse mecanismo consulta. HEARTBEAT.md = COMO ele vigia. hot.md = O QUE ele vigia. A9 cobre o mecanismo · essa seção cobre o contexto.

Aprofundamento do Bloco 3

Sessão · sintomas + sessions.json.

Memória de sessão é o chat atual. Janela enorme (200k tokens GPT-5.5 ≈ 150 páginas), mas finita. Quando enche, surgem sintomas de saturação:

Sintomas de contexto saturado

✓ Agente lento — demora pra responder, parece "pensando demais"

✓ Não responde — trava no meio, pula, ou retorna erro

✓ Alucina — inventa contexto que não existiu, mistura conversas

✓ Perde contexto antigo — esquece o que foi falado há 30min

Causa comum (amarração A4)

Em 90% dos casos: muitos tópicos no Telegram com o mesmo agente. Cada tópico cria um arquivo de transcript separado. Soma fica grande, contexto satura. 

~/.openclaw/agents/<agentId>/sessions/
├── sessions.json                              ← mapa de todas as sessões
├── <sessionId>.jsonl                          ← transcript da sessão
└── <sessionId>-topic-<threadId>.jsonl         ← 1 arquivo por tópico

Reforça regra A4: máximo 3-4 tópicos por agente no Telegram. Mais que isso, sessão satura rápido. Tópicos por tema (não por dia, não por humor) são a divisão saudável.

Tip prático ao vivo — diagnóstico

Se agente fica lento, pede pra ele rodar diagnóstico:

Prompt · diagnóstico de contexto cola no Telegram
roda /status e me diz como tá o uso de contexto. Se tá perto do limite (>70%), me sugere: 1. O que vale salvar antes de compactar 2. Se vale começar nova sessão (/new) ou compactar (/compact) 3. Se algum tópico tá inflando demais
💡 Use sempre que sentir lentidão. Em 80% dos casos a resposta é "vai precisar de /new em breve, salva isso e isso antes".

Quer ver o tamanho físico? Pede:

Prompt · ver tamanho do session store cola no Telegram
olha o tamanho do sessions.json em ~/.openclaw/agents/<id>/sessions/ e me diz se tá grande. Lista também os arquivos -topic-*.jsonl ordenados por tamanho — quero ver se algum tópico tá inflando.
💡 Path correto: sessions.json (plural). Cada tópico do Telegram vira arquivo separado <sessionId>-topic-<threadId>.jsonl.

Higiene = não esquecer de compactar + começar nova sessão. Bloco 5 cobre os comandos exatos. Princípio: salve antes de limpar (sempre).

Aprofundamento do Bloco 4

Memória semântica — busca por significado.

Sistema que faz o agente buscar pelo significado do que você perguntou, não pela palavra exata. Você pergunta "lembra do que decidimos sobre WhatsApp?" e ele acha decisoes/2026-04.md mesmo que o arquivo não tenha a palavra "WhatsApp" no nome.

Como funciona (alto nível)

1

Indexação contínua

Agente roda embeddings de cada arquivo do workspace (memory, content, archive) e guarda os vetores. OpenAI API key da A1 cobre.

2

Você pergunta com significado

"o que decidimos sobre comunidade?" · agente vetoriza a pergunta e busca os arquivos com vetores próximos.

3

Retorna resultados ranqueados

Top arquivos por proximidade semântica. Agente lê os mais relevantes, sintetiza resposta com data + raciocínio + próximos passos.

Demo concreto

Prompt · busca semântica cola no Telegram
lembra do que decidimos sobre WhatsApp? Faz busca semântica no workspace, não busca por palavra exata. Me responde com: 1. Data da decisão 2. Raciocínio (por que decidimos isso) 3. Onde tá registrado (arquivo) 4. Se mudou desde então
💡 Funciona pra qualquer pergunta sobre passado: pessoas, projetos, decisões, contexto. Quanto mais específico você for, melhor a busca acha.

Comandos manuais (Tier 2)

Pra power users e debug. Agente faz tudo automaticamente — você roda manual só se quiser debug ou índice forçado:

Terminal Managed comandos Tier 2
$openclaw memory status # estado do índice · quantos arquivos · última atualização $openclaw memory search "query" # busca semântica · retorna arquivos ranqueados $openclaw memory index --force # reindexar workspace inteiro · custa centavos
▶ Use --force só quando suspeita que índice tá desatualizado (ex: você moveu muitos arquivos manualmente · improvável no fluxo normal).

Custo

✓ Indexação completa: centavos pra workspace típico (poucos MB de markdown)

✓ Busca individual: fração de centavo (1 query semântica ≈ R$ 0,01-0,02)

✓ Reindexação automática: agente faz incremental (só arquivos modificados)

Princípio: "Sempre que você pesquisar sobre alguém, empresa, projeto — peça com detalhes. Quanto mais específico, melhor a busca semântica acha." Pergunta vaga = resposta vaga. Pergunta detalhada = resposta cirúrgica.

Aprofundamento do Bloco 5

Higiene de contexto — Claude Code vs OpenClaw.

Coração da aula. Aluno que veio do Claude Code reconhece quase tudo — OpenClaw seguiu padrão similar. Pra quem não veio, é introdução natural.

Aviso de abertura: "Se você usa Claude Code, vai reconhecer quase tudo. OpenClaw seguiu padrão similar." Comandos com mesma cara, comportamento próximo, bonus em alguns casos.

Comparação lado a lado

Movimento Claude Code OpenClaw
Compactar contexto /compact /compact Focus on [tema] (igual + flag de foco)
Nova sessão /new /new ou /reset
Trocar modelo na nova — /new <model> (bônus OpenClaw)
Ver estado /status /status (igual)
Salvar antes de limpar /salvar (skill) Comando natural: "salva tudo importante e esquece o resto"

Princípios da higiene OpenClaw

✓ Princípio absoluto (recap A0): "Salve antes de limpar." Memory Flush automático ajuda mas não é 100%

✓ Auto-compaction ON por default — dispara quando contexto enche ou modelo retorna context-overflow

✓ Comando natural cobre 80% dos casos — você não decora atalho · pede pro agente em português

✓ Quando mudar de assunto: dá /new pra começar limpo. Antes disso, salve

Comando natural — o jeito OpenClaw

Você não precisa decorar slash commands. Pede em português:

Prompt · salvar antes de limpar (natural) cola no Telegram antes de /new
salva tudo importante dessa conversa e esquece o resto. Em específico, quero garantir que vai pra memória persistente: 1. Decisões que tomamos (com data + raciocínio) 2. Pendências novas que surgiram 3. Contexto sobre pessoas/projetos que valha guardar 4. Próximos passos definidos Depois eu vou abrir nova sessão pra mudar de assunto.
💡 Frase-âncora pra memorizar: "salve antes de limpar". Sempre que for dar /new, roda esse prompt antes.

Comandos slash · referência rápida

Slash commands no Telegram manda direto na conversa
/status # mostra uso de contexto, sessão atual, modelo ativo /compact Focus on [tema] # compacta contexto mantendo foco em tema específico /new # nova sessão limpa /new gpt-5.5 # nova sessão trocando modelo (bônus OpenClaw) /reset # sinônimo de /new
▶ Slash commands rodam direto no chat (Telegram, Gateway). Não precisa abrir terminal.

Manutenção Tier 2 (terminal)

Comando opcional pra power users — limpa transcripts órfãos quando você acumulou muitas sessões antigas:

Terminal Managed limpeza opcional
$openclaw sessions cleanup --enforce # limpa sessões antigas + transcripts órfãos # reduz tamanho do session store (~/.openclaw/agents/<id>/sessions/)
▶ Use só se notar que disco tá enchendo. No uso PME normal, raramente é necessário.

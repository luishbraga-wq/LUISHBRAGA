Bloco 4 · Tier 1 (essenciais)

Os 5 comandos que seu agente já usa.

Inversão importante: esses comandos não são o seu trabalho do dia a dia. Quem roda eles é o seu agente — ele faz openclaw doctor toda segunda 9h via cron de health check (Bloco 7), aplica fixes seguros sozinho, e te avisa pelo Telegram só quando precisa de você. Você só conhece esses 5 pra 2 momentos: (1) curiosidade — ver o que ele tá fazendo · (2) emergência — quando ele tá com problema e você precisa diagnosticar manual. Decorar não é necessário — cheatsheet completo tá no Kit.

Default operacional: agente cuida do próprio ambiente. Cron semanal + auto-fix + alerta no Telegram = você só age quando ele te chama. Os comandos abaixo são fallback manual — não rotina.

1. openclaw status — vê se tá tudo OK

Comando padrão de "tá tudo bem aí?". Mostra estado dos módulos críticos (gateway, modelos, canais, skills) num resumo de 1 tela. É o primeiro comando que você roda quando algo parece estranho.

Terminal Managed cola e dá Enter
$openclaw status # esperado: gateway OK, model provider OK, channels OK # se algo aparecer com [WARN] ou [ERROR], roda doctor em seguida
▶ Usa quando: sentiu algo travado, agente não respondeu, ou simplesmente quer confirmação rápida que tudo está saudável antes de operar.

2. openclaw doctor — diagnóstico

Vai mais fundo que status. Lista tudo que está quebrado ou meio quebrado, com explicação do problema. Não conserta nada — só diagnostica. É o segundo comando da sequência status → doctor → doctor --fix.

Terminal Managed cola e dá Enter
$openclaw doctor # lista issues categorizadas: critical / warning / info # cada item mostra: o que tá quebrado + sugestão de fix
▶ Resultado típico: lista de issues com sugestão pra cada um. Lê com calma antes de pular pro --fix.

3. openclaw doctor --fix — auto-conserta

Aplica os fixes seguros automaticamente. Faz backup do config antes — se algo der errado, dá pra reverter. Não tenta consertar coisas sensíveis sem aviso (token revogado, credencial expirada). Pra isso, agente humano (você) decide.

Terminal Managed cola e dá Enter
$openclaw doctor --fix # aplica remediações seguras automaticamente # mostra: o que foi corrigido + o que precisa de ação manual # config anterior salvo em backup automático antes do fix
▶ Reverter: o backup do config fica salvo automaticamente. Se algo virou pior depois do fix, manda pro agente: "reverte o último doctor --fix" e ele restaura.

4. openclaw agent --message "..." — fala com agente direto

Manda mensagem pro agente sem precisar abrir Gateway/Telegram. Útil em scripts, debug, ou quando você quer testar uma resposta direto do terminal. Saída sai inline no próprio terminal.

Terminal Managed cola e dá Enter
$openclaw agent --message "tá tudo bem aí? roda doctor e me explica o resultado" # agente recebe a mensagem, executa doctor internamente, # e responde no terminal com o diagnóstico em português
▶ Caso de uso favorito: pedir pro agente diagnosticar problema sem você ler raw output. "explica em português o que tá errado e o que devo fazer".

5. openclaw exec-policy preset yolo — autopilot

Libera o agente pra executar qualquer comando dentro do Docker dele. NÃO vira root do servidor — só mexe em workspace, config, skills, credenciais. Como tudo está containerizado, risco real é baixo. Trade-off: mais autonomia × menos controle granular.

Terminal Managed cola e dá Enter (opcional)
$openclaw exec-policy preset yolo # libera autonomia operacional do agente dentro do container # reverter: openclaw exec-policy preset conservative
▶ Recomendação: liga se você quer agente resolvendo coisas sozinho (criar arquivo, instalar skill, ajustar cron). Pula se prefere aprovar caso a caso. Pode ligar depois — não é decisão definitiva.

Autopilot framing (do Bloco 5 da aula): "yolo libera o agente pra executar qualquer comando dentro do Docker. Ele NÃO vira root do servidor — só mexe em workspace, config, skills e credenciais. Como tudo está containerizado, risco real é baixo. Se preferir começar conservador, pula esse passo. Liga depois."

Bloco 4 · Tier 2 (úteis)

Os 5 comandos que aparecem na aula com menção rápida.

Esses 5 ficam na aula só com 1 frase de explicação cada — a sintaxe completa, flags e exemplos avançados estão no cheatsheet do kit. Reconhece o nome, sabe quando vai precisar.

Comando Pra que serve Quando usa
openclaw devices list Lista mac/mobile/browser conectados ao seu agente Auditar quem fala com agente
openclaw devices approve <id> Aprova device novo (ex: novo celular) Adicionar dispositivo à allowlist
openclaw security audit Análise de segurança · permissões, tokens, exposição Mensal · ou antes de conectar coisa nova
openclaw security audit --fix Aplica remediações seguras de segurança Após audit identificar issues conhecidas
openclaw config file Mostra o caminho do arquivo de config atual Quando agente pede pra editar config manualmente
openclaw config get <key> Lê valor de uma chave específica do config Debug · validar configuração
openclaw config set <key> <val> Define valor de uma chave do config Mudança pontual sem editar arquivo
openclaw restart Reinicia gateway · aplica mudanças pendentes Após mudança de config que pede reload

Princípio do curso: "seu agente lê documentação". Se travar em algum desses, manda pro agente "explica como funciona openclaw security audit e me dá um exemplo" — ele consulta a doc oficial e te ensina.

Bloco 6 · Tranquilidade Managed

Por que você não precisa se preocupar com update.

Aluno PME geralmente chega aqui com medo de "atualizar e quebrar tudo". No Managed, esse medo é desnecessário — Hostinger só serve versões estáveis do OpenClaw. Eles testam antes. Quando uma atualização chega no seu painel, é porque já passou pelo filtro deles.

NÃO rode openclaw update sem motivo. O comando funciona, mas bypassa o filtro de estáveis da Hostinger. Você pode pegar uma versão broken que ainda não foi promovida. Default é estável — deixa quieto.

Se algo travar pós-update

Pode acontecer (raro, mas acontece) que você ou a Hostinger empurrem nova versão e algo quebre. Rollback é via CLI: openclaw update --tag <versão-anterior>. O cheatsheet comandos-completos-openclaw.md cobre o fluxo completo de rollback.

Boa notícia bônus: Hostinger está lançando atualização do painel Managed com snapshot/backup nativo — restore em 1 clique pelo painel. Quando essa feature ativar, esse material vai ser atualizado pra cobrir o fluxo. Por enquanto, snapshot fica via comando + skill backup-workspace-github (próximo passo do curso, ativada via Item Integrações da CHECKLIST do Kit).

Princípio que entra aqui: Managed te protege do que dá pra automatizar (testes de versão, isolamento Docker, rede), mas você fica responsável pelo que só humano decide — quando ligar autopilot, quando fazer snapshot manual, quando aprovar device novo. Boas notícias por enquanto.

Bloco 7 · Coração da aula ⭐

Cron semanal de health check.

Esse é o entregável mais forte da A2. Em vez de você lembrar de checar saúde do agente, ele checa sozinho toda segunda às 09:00 BRT e te avisa pelo Telegram. Se tudo OK, manda 1 linha. Se tiver issue, detalha e sugere fix. Você só age quando ele te chama.

Princípio reforçado: "Seu agente cuida do próprio ambiente." Versão Managed: monitora + avisa, você decide quando agir. Versão VPS (R1-R4): monitora + aplica fixes seguros automaticamente. Aqui no curso MAIN, ficamos no Managed-first.

Prompt pro agente criar o cron

Cola esse prompt no Telegram (ou Gateway, mas Telegram é melhor pra ver resultado depois). O agente lê, entende, propõe configuração, pede confirmação, e cria o cron. Não precisa mexer em arquivo nenhum.

Prompt #1 · cron de health check semanal cola no Telegram (ou Gateway)
Cria um cron semanal (toda segunda 09:00 BRT) que: 1. Roda `openclaw doctor` e me manda o resultado pelo Telegram (tópico Operação) 2. Se tudo OK: 1 linha "tudo certo, agente saudável" 3. Se tiver issue: detalha o que tá errado e sugere fix 4. Lembra: "se passou de 30 dias sem backup do workspace pro GitHub, roda skill backup-workspace-github" (snapshot Hostinger nativo no painel chega numa atualização futura — quando vier, atualizo este prompt pra mencionar restore em 1 clique pelo painel) NUNCA aplicar fix automaticamente em coisa sensível — só me avisar. Confirma comigo antes de criar o cron.
💡 Resultado: agente confirma o plano, mostra como vai ficar a mensagem de exemplo, e cria o cron. Próxima segunda 09:00 BRT você recebe o primeiro health check no Telegram.

Por que o tópico Operação

Adiantei pra você: na A3 (próxima aula) você vai aprender a estruturar Telegram com 4 tópicos por tema — Conteúdo, Operação, Aprendizado, Financeiro. Health check é assunto de Operação. Esse cron já tá pronto pra postar no tópico certo quando você criar — ou no DM, se preferir começar simples e migrar depois.

Aluno que ainda não viu A3: tudo bem. O agente vai postar no DM padrão do Telegram por enquanto. Quando você criar o supergrupo + tópicos na A3, é só pedir "move o cron de health check pra postar no tópico Operação" que ele ajusta sem recriar nada.

Vitória visível: próxima segunda 09:00 BRT, mensagem do agente chega no seu Telegram com o status. Você não precisou lembrar. Não precisou abrir o painel. Não precisou rodar comando. Esse é o tipo de relacionamento que o curso constrói com seu agente.

Bloco 8 · Plano B

Se algo quebrar.

Em ordem do mais comum pro mais raro. Pra cada cenário, ação clara — sem você ter que pesquisar.

1

Algo soa estranho · agente lento ou meio confuso

Roda openclaw status · se aparecer warning, vai pra openclaw doctor · aplica openclaw doctor --fix se as issues forem seguras (auto-corrigíveis). Resolve 80% dos casos sem você precisar de mais nada.

2

Doctor não resolve · agente não responde

Roda openclaw restart · reinicia gateway. 99% dos travamentos somem aqui. Se ainda assim não voltar, vai pro próximo passo.

3

Restart não resolve · suspeita que é problema de versão

Cheatsheet do kit cobre rollback: openclaw update --tag <versão-anterior>. Volta pra última versão estável que você sabia que funcionava. Anota qual versão você tava antes em algum lugar — vale a pena.

4

Restore via snapshot

Painel Hostinger → seu OpenClaw → área de backups/snapshots → restore. Volta pro estado anterior. Snapshot nativo do painel chega numa atualização futura — atualmente snapshots ficam via comando + skill backup-workspace-github do kit.

5

Última saída · suporte Hostinger

Painel Hostinger tem chat com agente da Hostinger. Resposta <5min. É a saída pra quando algo do servidor quebrou (não do OpenClaw). Cobre 100% dos casos que você não consegue resolver no terminal.

Limite Managed: 1 OpenClaw por plano. Quer rodar um segundo agente independente (ex: agente de cliente diferente, ou separar pessoal × trabalho)? Caminhos: upgrade pro plano superior da Hostinger OU bônus VPS standalone (R1-R4 do curso). MAIN cobre 1 OpenClaw multi-agente — vários papéis no mesmo workspace.
